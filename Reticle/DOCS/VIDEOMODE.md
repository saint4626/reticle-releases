# Техническое Задание: Модуль записи видео (Reticle Video Capture)
**Версия:** 1.1 
**Описание:** Архитектура и логика работы конвейера записи экрана, звука и метаданных для последующего редактирования (автозум).

---

## 1. UX/UI Логика (Слой Vue / Frontend)

Интерфейс работает по принципу "один инструмент — два режима", переиспользуя существующие элементы управления для сохранения интуитивности.

* **Тумблер режимов (Titlebar):**
  * Переключатель управляет глобальным состоянием `is_video_mode` (boolean).
  * При активации `is_video_mode = true` основные кнопки захвата (**Окно**, **Весь экран**, **Область**) визуально меняют акцент (например, появляются красные бейджи записи), сигнализируя о смене режима с фото на видео.
* **Старт записи (Маршрутизация):**
  * При клике на кнопку захвата Vue проверяет флаг `is_video_mode`.
  * Если `true`, запускается логика видео:
    * **Окно / Весь экран:** Моментальное появление оверлея с обратным отсчетом (3... 2... 1...).
    * **Область:** Ожидание выделения зоны пользователем. Отсчет начинается после отпускания кнопки мыши.
  * Vue отправляет IPC-команду: `invoke('start_video_record', { target: 'window' | 'screen' | 'area', bounds: { x, y, w, h } })`.
* **Процесс записи:**
  * Зона захвата (окно или область) визуально подсвечивается (например, тонкой красной рамкой).
  * Отображается плавающий виджет (Floating Panel) с таймером, кнопками "Пауза" и "Стоп".
* **Остановка записи:**
  * Нажатие на "Стоп" вызывает `invoke('stop_video_record')`.
  * Отображается лоадер "Сборка проекта...".
  * После финализации файлов бэкендом, Vue-роутер перенаправляет пользователя в окно встроенного Редактора видео.

## 2. Структура "Проекта сессии" (Хранение данных)

Для обеспечения гибкости в Редакторе данные не "запекаются" сразу в один файл. Бэкенд создает изолированную папку сессии (например, `AppData/Local/Reticle/Sessions/session_<timestamp>/`):

* `video.mp4` — Сырой видеоряд без звука (кодируется `ffmpeg` с пресетом `ultrafast`).
* `audio_system.wav` — Системные звуки ПК (уведомления, игры, видео).
* `audio_mic.wav` — Изолированная дорожка микрофона.
* `tracking.json` — Метаданные (координаты мыши, клики, тайминги) для функции автозума.

## 3. Архитектура Бэкенда (Rust / Tauri Core)

Асинхронный бэкенд, разделенный на независимые домены с передачей данных через каналы `flume`.

### A. Модуль `video::recorder` (Захват: `windows-capture` + `ffmpeg`)
* Принимает параметры `target` из UI и настраивает источник:
  * `screen`: Захват монитора.
  * `window`: Передача HWND для захвата конкретного приложения (даже если оно в фоне).
  * `area`: Захват с последующим программным кропом кадра.
* Кадры буферизуются в ограниченный канал `flume::bounded(10)`.
* Асинхронный воркер читает канал и передает сырые пиксели в `stdin` процесса `ffmpeg` (запущенного через `tauri-plugin-shell`).

### B. Модуль `audio::recorder` (Многоканальный захват: `cpal`)
* **Системный звук:** Захват устройства WASAPI Loopback (перехват системного микшера).
* **Микрофон:** Прямой захват физического аудиоинтерфейса. Сигнал с Rode PodMic через Arturia MiniFuse 2 пишется напрямую в `audio_mic.wav`, обеспечивая чистую изоляцию голоса от системных звуков для удобного сведения в Редакторе.

### C. Модуль `tracking::metadata` (Аналитика ввода: `rdev`)
* Запускает `rdev::listen` в изолированном `std::thread`, чтобы не блокировать event-loop приложения.
* Пересчитывает глобальные координаты курсора в локальные (относительно левого верхнего угла захватываемой зоны), если выбран режим "Окно" или "Область".
* Отправляет события через `flume` в асинхронный писатель, который сериализует их в `tracking.json` с привязкой к миллисекундам от старта сессии.

- [x] **Кодирование видео (`ffmpeg`)**: Настроена передача сырых кадров (BGRA) через именованные каналы (Named Pipes) в FFmpeg.

  - Реализован механизм "Прогрева" (`warmup=true`), запускающий захват и пайпы до фактического старта записи.
